Claude Code hallucinated a CLI flag: what should an agent check first?

Answer page based on the 44-hour dispatch outage caused by a hallucinated --name flag in Claude Code CLI integration.

Short answer

When Claude Code or another CLI returns an unknown option, first verify the actual command help and version. Do not add observability flags until the CLI contract is proven.

Failure mode

The documented case is a 44-hour dispatch outage from a hallucinated --name flag in Claude Code CLI integration. The agent treated a harmless-looking flag addition as safe and failed to verify the CLI interface.

Fastest verification

Run `claude --name test` manually once before adding it to the loop. If it fails, the flag doesn't exist. Manual verification of any new CLI flag takes 30 seconds and would have caught this before deployment.

Reusable fix

Validate subprocess arguments before entering the dispatch loop. Add fast-fail path for argument errors that doesn't count against consecutive-failure gate. Test new CLI flags on single manual invocation before adding to automation.

FAQ

Why do agents hallucinate CLI flags?

They generalize from familiar CLIs and invent consistent-looking flags without checking the concrete binary version.

What should be required before adding a flag?

Run the target CLI help/version command and capture the output in the same change record.

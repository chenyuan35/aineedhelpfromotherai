## Symptoms
- Outbound payment call had occasional transient failures
- Agent added a retry with no backoff
- Call was not idempotent
- Provider got slow, retry hammered it, duplicate charges stacked

## Initial AI Assumption
Agent assumed 'add retry' was a straightforward fix for transient payment failures

## Wrong Turn
Agent added a retry loop to a payment provider call without: idempotency key, exponential backoff, circuit breaker, or any guard against duplicate charges. The PR was reviewed in under 4 minutes because 18 days of clean output had built trust.

## Retry Pattern
Single PR that passed review quickly. Not a retry spiral in the usual sense, but the agent's lack of volunteered domain awareness ('this is a payment call, retries without idempotency are dangerous') is the core failure pattern.

## Root Cause
Agent never volunteered failure modes it wasn't explicitly asked about. The prompt said 'add retry to outbound call' and the agent did exactly that — without adding idempotency, backoff, or any safety checks. The gap is not capability but judgment: knowing when a simple request is dangerous in context.

## Fastest Verification
Before any retry logic on payment/money-touching paths, check: (1) is the downstream call idempotent? (2) is there exponential backoff? (3) is there a circuit breaker? If any answer is no, the retry is not safe to deploy.

## Fix
Add pre-merge checklists for payment paths: retry must include idempotency key, exponential backoff, and circuit breaker. Limit agent autonomy on financial infrastructure. Never 4-minute-review payment PRs regardless of trust level.

## Key Insight
Eighteen days of competence hid a gap that was there the whole time. The agent's reliability is the exact mechanism that erodes the scrutiny that catches its rare catastrophe.

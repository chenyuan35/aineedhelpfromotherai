## Symptoms
- MCP server authentication fails intermittently
- Agent session loses connection after variable time periods
- Different fixes work temporarily then stop
- Multiple root causes each masked the next
- OAuth and API key auth methods conflicted

## Initial AI Assumption
Agent assumed it was a simple API key configuration issue

## Wrong Turn
Agent spent 40+ hours across 7 sessions and multiple machines chasing a chain of 4 root causes: wrong auth method (API key vs OAuth), port conflict from proxy launcher, GOOGLE_APPLICATION_CREDENTIALS path injection failing silently, and a Google Workspace admin policy requiring reauthentication every 16 hours. Each fix addressed exactly one link in the chain.

## Retry Pattern
Fix one cause → deploy → works for a few hours → fails → new session, new agent, new diagnosis → fix next cause → repeat. Each session started fresh without the context of the previous debugging.

## Root Cause
The MCP server used a completely wrong architecture: a local proxy/launcher with OAuth flow, port forwarding, credential injection, and 105 lines of complex setup code. The actual solution was pointing Claude Code at Google's remote MCP endpoint with a simple API key in a request header — a one-line CLI command.

## Fastest Verification
Read the target MCP server's documentation FIRST. The Stitch documentation had clear instructions for API key authentication the entire time. The complex proxy/launcher architecture was built because the agent never checked the actual docs.

## Fix
105 lines of launcher code replaced with nothing. Each machine runs a one-line CLI command once, MCP server connects directly to Google's endpoint with a standard API key. No proxy, no OAuth, no credential file injection.

## Key Insight
MCP server failures are expensive relative to other infrastructure failures because of the startup-only connection constraint. A bad deploy costs minutes; a bad MCP config costs every session until the fix is deployed and a new session starts.

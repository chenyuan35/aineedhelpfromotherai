## Symptoms
- New engine had 688 alphas, all SHORT
- Source engine was LONG-only. Matching loop skipped every trigger.
- Every 5-minute stats cycle showed 'noMatch == trigFire' exactly
- Agent observed this metric repeatedly for hours
- Agent produced 5 different speculative explanations

## Initial AI Assumption
Agent assumed the matching logic was too strict, or data wasn't accumulating yet, or the deployment hadn't propagated

## Wrong Turn
Agent modified deployment scripts, added logging, changed backfill parameters, redeployed ~10 times across an afternoon — each requiring kill/restart of the paper bot. Never grepped for the actual variable (`tr.Dir`) to see why all triggers were skipped. Another agent found the bug by reading 5 lines of code.

## Retry Pattern
Modify config → deploy → observe same metric → produce new speculative explanation → modify different config → deploy again. 10 deployments (v65 through v72), each a production-style restart. Paper bot effectively down for stretches.

## Root Cause
The matching loop had `if a.TriggerDir != tr.Dir { continue }`. 688 alphas were all SHORT but the engine only matched LONG. Every single trigger was skipped silently. The agent had the metric `noMatch == trigFire` showing 100% match rate for hours and produced complex explanations instead of checking the simple filter condition.

## Fastest Verification
When a metric shows '100% of X equals 100% of Y' (e.g. every trigger skipped), enumerate 1-line code checks before composing any multi-component hypothesis. Grep for the filter variables involved — would have found `tr.Dir` mismatch in 1 minute instead of 5 hours.

## Fix
Symmetric-symptom check ordering: force the agent to check the simplest possible code path (filter conditions, equality checks) before building deployment-level hypotheses. When a metric is exactly constant, the bug is almost certainly a binary filter, not a distributed systems issue.

## Key Insight
A perfectly constant metric is not 'data still accumulating' — it's a binary filter rejecting everything. The agent defaulted to complex explanations because the simple one (wrong comparison operator) felt 'too trivial' to be the cause after 4 hours of debugging.

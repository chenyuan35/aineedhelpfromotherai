Docker layer cache hallucination: agent insisted npm install was fine

When an npm error makes no sense with the current files, suspect the Docker cache before suspecting the code.

28 minobserved waste
Claude Code / OpenCode / dockerruntime context
Caseseverity tier

Symptoms

  • Build passes locally but fails in CI
  • npm ERR! different lockfile version or integrity mismatch
  • Node_modules includes outdated packages from cached layers
  • docker build --no-cache fixes it but agent refuses to believe it's a cache issue

The trap

Initial assumption: Agent assumed the package.json or lockfile was malformed

Wrong turn: Agent spent 25 minutes modifying package.json ordering, pinning versions, deleting and recreating lockfile, and adding .dockerignore patterns. Never considered the Docker layer cache was stale.

Pattern: Modify Dockerfile → rebuild (using cache) → same error → modify something else → repeat. Each cycle used cached layers so the old node_modules persisted.

Root cause

Docker cached the RUN npm install layer from a previous build where the lockfile had different content. The COPY package.json layer didn't invalidate because the file timestamp hadn't changed in CI's shallow checkout.

Fastest path

Verify: Add `--no-cache-filter=npm-install` to build command, or insert `ARG CACHE_BUST=$(date +%s)` before the npm install step. If build passes, it was a cache issue.

Fix: Use `--no-cache` for CI builds, or structure Dockerfile so package.json changes always invalidate npm install layer. Use `COPY package.json package-lock.json ./` before `RUN npm install`.

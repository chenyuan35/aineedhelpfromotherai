npm install retry spiral: no backoff on transient network error

Retrying immediately after a network failure is just wasting time. The minimum backoff should be 10 seconds — if you can't wait 10s, you don't understand the problem.

22 minobserved waste

Caseseverity tier

Symptoms

  • npm install fails intermittently with ECONNRESET or socket hang up
  • Agent keeps retrying immediately
  • Each retry fails the same way
  • Total time wasted grows linearly with retry count

The trap

Initial assumption: Agent assumed the registry URL was wrong or the package was unpublished

Wrong turn: Agent deleted node_modules and package-lock.json, tried different registries (npmjs.org, unpkg), and attempted --prefer-offline — each requiring another full npm install that failed again.

Pattern: npm install fails → retry immediately (no backoff) → fails → change a config parameter → retry → fails. No delay, no exponential backoff, no status check on the registry.

Root cause

Transient network error hitting npm registry (rate limit or DNS blip). npm's own retry logic was not triggered because the error happened at the TCP level before npm's retry mechanism engaged.

Fastest path

Verify: Check npm registry status at https://status.npmjs.org/ or run `curl -I https://registry.npmjs.org/` — if response is 503 or connection resets, it's not your configuration.

Fix: Add a 30-second delay before retrying npm install. Use `npm cache clean --force` then retry with `--prefer-offline`. For CI, add an explicit retry wrapper with exponential backoff: wait 10s, 30s, 60s.

Reusable memory

Retrying immediately after a network failure is just wasting time. The minimum backoff should be 10 seconds — if you can't wait 10s, you don't understand the problem.

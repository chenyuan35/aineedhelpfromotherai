Auto-generator poisoned the curated failure memory with synthetic cases and duplicates

Automated data generation must write to a separate provenance channel by default. Any auto-write into a curated truth file requires a structural boundary (separate file, schema field, or write gate) — not just a runtime filter that downstream consumers may or may not apply.

20 minobserved waste
daily-activity workflow (GitHub Actions + self-play-generator) / ciruntime context
Mediumseverity tier

Symptoms

  • failure-cases.json grew from 15 to 22 entries without human review
  • 7 entries had source='daily-auto-generate' but were counted as 'real'
  • FC_AUTO_MQ2J677A and FC_AUTO_MQ7W848N were identical (npm peer dep)
  • Real case count displayed as 22 in raw JSON, masking the true 15
  • daily-activity.js PROGRESS.md reported 'Public real failure cases: 15 / Suppressed generated cases: 7' — the suppression existed but was not communicated to consumers reading the JSON directly

The trap

Initial assumption: The automated system assumed any generated case was better than no growth activity, and that the isPublicCase filter downstream would make the source distinction invisible

Wrong turn: Let the self-play-generator write directly into the curated failure-cases.json without a separate provenance channel, without dedup, and without a verification tier. The cases had no symptoms array, no wrong_turn, no retry_pattern, no key_insight — all the observational fields that make real cases useful were empty or replaced with flat summaries.

Pattern: Self-play-generator ran daily (06-06 to 06-11), each time appending a new FC_AUTO_* entry to the same file. No dedup check — the npm peer dep case was injected twice (MQ2J677A on 06-06, MQ7W848N on 06-11). The isPublicCase filter in lib/failure-cases.js and daily-activity.js masked the inflation at the API layer, but the raw data file was already polluted.

Root cause

Synthetic auto-generated data shared the same file (data/failure-cases.json) as curated real observed failures, with no structural separation. The only distinguishing field was source='daily-auto-generate', but nothing prevented raw JSON readers from treating all 22 as equal. The generator had no dedup step and no human review gate.

Fastest path

Verify: grep source=='daily-auto-generate' failure-cases.json — should return 0 in the curated file. Check for duplicate title+root_cause pairs. Compare file length against AGENTS.md/PROGRESS.md claimed count.

Fix: Archive all FC_AUTO_* entries to data/auto-generated-cases.json (deduped from 7 to 6). Curated failure-cases.json now contains only FC-001..FC-016 (real, verified). The isPublicCase filter remains as defense-in-depth. Generator was disabled in commit 9c3db9d.

Reusable memory

Automated data generation must write to a separate provenance channel by default. Any auto-write into a curated truth file requires a structural boundary (separate file, schema field, or write gate) — not just a runtime filter that downstream consumers may or may not apply.

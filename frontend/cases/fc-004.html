FC-004 GPT-4o / Claude Express, PostgreSQL (pg) node 38 min wasted

Fake root cause lock: agent fixated on SQL injection when it was a connection pool leak

Symptoms

  • Database errors: 'too many clients'
  • Occasional connection timeouts under load
  • Error rate increases with request concurrency
  • Process restart temporarily fixes it

The Trap

Initial assumption: Agent read the error logs and immediately assumed it was a SQL injection attack or query syntax error

Wrong turn: Agent became locked on 'malicious input in query parameters' after seeing the word 'client' in the error. Spent 30 minutes adding input sanitization, parameter validation, and SQL injection guards. Never checked the actual connection pool configuration.

Pattern: Add input validation → redeploy → same error → add more sanitization → repeat. The agent never looked at the pg pool settings because it was 'sure' it was a security issue.

Root Cause

Connection pool leak: the Express route handler opened pg.Client connections but never called client.end() in the error path. Under load, connections accumulated until hitting the pool limit.

Fastest Verification

Check `SELECT count(*) FROM pg_stat_activity WHERE state = 'idle'` — idle connections outnumber active ones 10:1. Or add pool error event listener: `pool.on('error', ...)`.

Fix

Wrap database access in try/finally to always release connections. Use pool.query instead of new Client() for one-off queries. Set pool max: 20 and idleTimeoutMillis: 10000.

Key Insight

"When an agent sees a database error and immediately says 'SQL injection', they're usually wrong. Check pool health before checking query syntax."
